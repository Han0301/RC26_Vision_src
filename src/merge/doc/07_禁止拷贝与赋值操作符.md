# `= delete` — 禁止拷贝构造与拷贝赋值

> 理解 `serial.h` 第 31~32 行：
> ```cpp
> Ten_serial(const Ten_serial& serial) = delete;
> Ten_serial& operator=(const Ten_serial& serial) = delete;
> ```

---

## 1. 逐词拆解

### 第 31 行：拷贝构造函数

```cpp
Ten_serial(const Ten_serial& serial) = delete;
```

| 语法部分 | 含义 |
|----------|------|
| `Ten_serial` | 函数名（和类名相同，是构造函数） |
| `(const Ten_serial& serial)` | 参数：接收一个 const 引用作为"源对象" |
| `= delete` | **C++11 特性**：显式删除此函数 |

如果没有 `= delete`，编译器会自动生成一个拷贝构造函数，作用是把源对象的所有成员逐个复制到新对象。

### 第 32 行：拷贝赋值运算符

```cpp
Ten_serial& operator=(const Ten_serial& serial) = delete;
```

| 语法部分 | 含义 |
|----------|------|
| `operator=` | 赋值运算符重载 |
| `Ten_serial&` | 返回自身引用（支持 `a = b = c` 链式赋值） |
| `(const Ten_serial& serial)` | 参数：右侧被赋值的对象 |
| `= delete` | 显式删除 |

---

## 2. `= delete` 做了什么

`= delete` 告诉编译器：

> **"这个函数不存在，任何人试图调用它，直接编译报错。"**

```cpp
// 因为有 = delete，以下代码全部编译失败 ❌

Ten::Ten_serial& s1 = Ten::Ten_serial::GetInstance();
Ten::Ten_serial& s2 = Ten::Ten_serial::GetInstance();

Ten_serial s3 = s1;      // ❌ 拷贝构造被 delete
s2 = s1;                  // ❌ 拷贝赋值被 delete
Ten_serial s4(s1);        // ❌ 拷贝构造被 delete

auto s5 = s1;             // ❌ 拷贝构造被 delete
```

### 对比 C++03 的旧做法

```cpp
// C++03 的做法（声明为 private 且不实现）
class Ten_serial {
private:
    Ten_serial(const Ten_serial&);       // 只声明不实现
    Ten_serial& operator=(const Ten_serial&);  // 只声明不实现
};
```

| 特性 | C++03 方式（private + 不实现） | C++11 `= delete` |
|------|:---:|:---:|
| 报错时机 | **链接时**才报错（undefined reference） | **编译时**直接报错 |
| 报错信息 | 晦涩难懂 | 清晰明确 |
| 友元类 | 可以绕过 private 访问，但链接失败 | 编译直接拒绝 |
| 表达能力 | 只能暗示"不应使用" | 明确宣告"禁止此操作" |

---

## 3. 为什么单例类要禁止拷贝？

### 场景：如果不禁止

```cpp
auto& s1 = Ten::Ten_serial::GetInstance();

// 假设没有 = delete，以下代码合法：
Ten_serial s2 = s1;  // 拷贝构造 → s2 是 s1 的副本
```

后果：

| 问题 | 说明 |
|------|------|
| **单例被打破** | 两个对象存在，不再是"唯一" |
| **串口资源冲突** | 两个对象中的 `serial_` 操作同一个底层 fd |
| **锁被复制** | `std::mutex` 不可复制，拷贝后行为未定义 |
| **状态不同步** | 一个对象修改了状态，另一个不知道 |

### 具体到成员

```cpp
serial::Serial serial_;   // 复制后两个对象操作同一个串口 → 混乱
std::mutex send_mtx_;     // mutex 不可复制（拷贝构造被删除）
std::mutex read_mtx_;     // mutex 不可复制
```

## 4. 三重保护：`Ten_serial` 的完整防护体系

```cpp
class Ten_serial
{
public:
    // ① 第31行：禁止拷贝构造
    Ten_serial(const Ten_serial& serial) = delete;
    
    // ② 第32行：禁止拷贝赋值
    Ten_serial& operator=(const Ten_serial& serial) = delete;

private:
    // ③ 第81行：构造函数私有，外部不能 new
    Ten_serial(const std::string& port, const size_t& serial_baud);
};
```

这三道防线共同保证了**绝不可能创建第二个实例**：

```cpp
// 所有非法尝试都编译失败
Ten_serial s(port, baud);           // ❌ private 构造函数
Ten_serial s2 = GetInstance();      // ❌ 拷贝构造被 delete
s2 = GetInstance();                  // ❌ 赋值被 delete
auto* p = new Ten_serial(port, baud); // ❌ private 构造函数
auto s3 = GetInstance();             // ❌ 拷贝构造被 delete

// 唯一合法的途径
auto& serial = Ten::Ten_serial::GetInstance();  // ✅
```

## 5. `= delete` 的其他妙用

`= delete` 不只用于单例，还可以：

### 禁止特定类型转换

```cpp
class Foo {
public:
    void func(int x);           // 接受 int
    void func(double) = delete; // 禁止 double → 防止隐式转换
};

Foo f;
f.func(42);     // ✅ 调用 func(int)
f.func(3.14);   // ❌ 编译错误，double 被 delete
```

### 禁止在堆上创建对象

```cpp
class StackOnly {
public:
    void* operator new(size_t) = delete;  // 禁止 new
};

StackOnly obj;   // ✅ 栈上可以
auto* p = new StackOnly();  // ❌ 编译错误
```

## 6. 对比：项目中还有哪些 `= delete`

在 `serial.h` 中，这两行删除了编译器**默认会生成**的两个特殊成员函数：

| 特殊成员函数 | 声明 | 状态 |
|:----------:|------|:---:|
| 默认构造函数 | 未声明 | **不存在**（编译器不生成，因为已有自定义构造函数） |
| 析构函数 | `~Ten_serial()` | **自定义**，关闭串口 |
| 拷贝构造函数 | `= delete` | **显式删除** |
| 拷贝赋值运算符 | `= delete` | **显式删除** |
| 移动构造函数 | 未声明 | **不存在**（因为拷贝构造被删除，编译器不生成） |
| 移动赋值运算符 | 未声明 | **不存在** |

---

## 7. 一句话总结

> **`= delete` 说人话就是：这个操作被拉黑了。**
>
> 单例类的对象不应该被复制——复制一个串口对象在语义上是荒谬的。C++11 的 `= delete` 让这种荒谬在编译期就被扼杀，而不是等到运行时出现诡异 bug 才发现。
