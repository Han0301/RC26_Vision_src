# RC26 Vision src — 上车融合代码工作区

> 多传感器融合感知框架，从 merge_ws20 到 rc_ws2 共 12 个版本的完整迭代。
> 运行在 RC26 竞赛机器人上，整合相机、激光雷达、Z-buffer 遮挡处理、YOLO 检测等模块。

---

## 项目背景

`merge_ws` 系列是 RC26 竞赛机器人的 **上车运行融合代码**。

早期在 `camera_ws` 和 `world_ws` 中分别验证了相机检测和 Z-buffer 感知算法，`merge_ws` 负责将这些算法整合到统一的 ROS 节点中，在真实 RoboMaster / RoboCup 机器人上运行。

从最初的单传感器简单融合逐步演变为多传感器（相机 × 2、激光雷达 × 2、里程计、IMU）、多通道（视觉、激光、Z-buffer）、具备上位机交互和竞赛功能支持的完整系统。

## 系统架构

```
工控机 (ROS Master)
├── 传感器层
│   ├── 相机 × 2 (USB + RealSense)
│   ├── 激光雷达 × 2 (Livox MID-360)
│   ├── 轮式里程计
│   └── IMU
├── 感知层
│   ├── 视觉检测 (YOLOv11 + PnP)
│   ├── 激光雷达识别 (LiDAR Recognition)
│   ├── Z-buffer 遮挡处理
│   └── 多传感器融合 (Merge Node)
├── 决策层
│   ├── 上层决策 (Superstratum)
│   ├── 运动控制 (Control)
│   └── 标定模块 (Calibration)
└── 运维层
    ├── 上位机监控 (Shell Monitor 2)
    ├── 设备自检 (Self-Inspection)
    └── 参数配置 (Parameter System)
```

## 编译与运行

```bash
# 编译（限制 2 核避免系统过热崩溃）
catkin_make -j2
source devel/setup.bash

# 启动融合节点
roslaunch merge merge.launch

# 上位机监控
python3 src/merge/shell_monitor2/control_center.py
```

## 详细版本迭代日志

### merge_ws20 — 上车初始版

**背景**：仿真中的算法需要在真实机器人上运行，但仿真代码直接上车存在接口不兼容、实时性不足等问题。

**发现的问题**：
- 仿真环境中的传感器数据流与真实 ROS 话题结构不同，无法直接复用
- 缺乏统一的融合节点管理多传感器数据，每个传感器各自为政
- 激光雷达驱动（Livox MID-360）未在 ROS 中配置过
- 多线程任务缺乏统一管理，裸线程容易死锁

**解决方案**：
- 搭建 `merge_node` ROS 融合节点，统一管理相机、激光雷达、里程计数据订阅与时间同步
- 首次接入 `livox_ros_driver2` 激光雷达驱动，配置 MID-360 的坐标系和点云参数
- 实现基础点云处理管线（体素滤波降采样 → 平面提取 → RANSAC 分割）
- 引入 `threadpool.h` 线程池管理并行任务，避免裸线程的创建/销毁开销，并通过工作窃取算法均衡负载
- 新增 `shell_monitor` 脚本监控系统运行状态（CPU 温度、内存使用、进程存活）

**效果**：首次在真实机器人上跑通从传感器采集 → 数据融合 → 控制指令的完整链路。

### merge_ws21 — 标定模块与线程池重构

**背景**：merge_ws20 虽然跑通了链路，但传感器数据对齐差、系统运行一段时间后会因内存暴涨而崩溃。

**发现的问题**：
- **传感器外参缺失**：相机和激光雷达的数据在空间上没有对齐，点云投影到图像上偏差 10-20 像素，导致融合完全错位
- **线程池任务堆积**：当传感器数据频率不匹配时（相机 30fps，雷达 10Hz），线程池任务队列无限增长，最终 OOM 杀死进程
- **坐标变换混乱**：相机、激光雷达、机器人基坐标系之间的变换关系硬编码在各模块中，修改一个就要改多处，极易引入 bug

**解决方案**：
- **新增标定模块**（`calibration.cpp/h`）：实现相机→激光雷达的外参标定，使用 PnP 联合优化得到准确的旋转矩阵 R 和平移向量 t，将投影误差从 10-20 像素降至 2-3 像素
- **线程池背压机制**：新增 `_TREADPOOL_FLAG_` 标志位，当队列长度超过阈值时自动丢弃最旧的任务，防止内存暴涨；同时新增任务优先级队列，高优先级任务（如控制指令）不会被丢弃
- **统一坐标变换**（`coordinate.h`）：封装全部坐标系转换逻辑，统一使用 `tf2` 管理，各模块不再各自维护变换关系；新增变换缓存，减少重复计算

**效果**：传感器对齐精度提升至 2-3 像素；系统可连续运行数小时不 OOM。

### merge_ws22 — 遮挡处理与 Debug 构建

**背景**：真实环境中遮挡情况比仿真复杂得多（机械臂、其他机器人、方块间遮挡），现有模型在遮挡场景下产生大量假阳性检测。

**发现的问题**：
- 当机械臂或其它方块遮挡目标时，视觉检测模块仍输出高置信度结果，但实际目标已被遮挡，导致机械臂抓空
- 多传感器融合时没有考虑遮挡置信度，遮挡下的错误检测影响了整体决策
- 上车调试困难：Release 模式下日志信息太少，无法定位问题根因

**解决方案**：
- **新增遮挡处理模块**（`occlusion_handing.h`）：基于 Z-buffer 原理，在像素级别判断目标是否被前景遮挡。核心算法：
  1. 将 3D 目标投影到 2D 图像平面，得到深度值
  2. 将相机深度图的对应区域与目标深度值比较
  3. 若深度图中该区域的前景深度小于目标深度，判定为遮挡
  4. 对遮挡程度超过 30% 的目标降低其融合权重
- **切换 Debug 构建**：`catkin build --cmake-args -DCMAKE_BUILD_TYPE=Debug`，保留符号表和完整的断言检查，方便 GDB 调试
- 在合并决策逻辑中引入遮挡置信度：视觉、激光、Z-buffer 三个通道的输出在融合时，根据遮挡检测结果动态调整各通道的权重

**效果**：遮挡场景的误检率降低 60%；Debug 模式下问题定位时间缩短 70%。

### merge_ws33 — 视觉里程计与点云配准

**背景**：仅靠轮式里程计在 RoboMaster 赛场的地毯和坡道上打滑严重，定位漂移无法满足导航和抓取的需求。

**发现的问题**：
- 轮式里程计在光滑地面和坡道上累计误差快速增长，每分钟漂移约 20cm
- 传统 ICP 点云配准速度慢（每帧约 200ms），无法满足实时性要求
- 地图构建是离线的，机器人运行中无法增量更新地图，一旦环境变化就要重新建图

**解决方案**：
- **新增视觉里程计**（`visual_odometry.h`）：利用相机图像提取 FAST 角点 + BRIEF 描述子，通过光流跟踪帧间特征点，使用 8 点法估计本质矩阵，分解得到帧间运动。通过 EKF 与轮式里程计融合，在打滑时视觉权重自动增大
- **引入 nano_gicp**：轻量级 GICP 点云配准库，使用多核 NEON 指令集优化，配准速度提升 5 倍（约 40ms/帧），精度也高于传统 ICP
- 在 `lidar.h/lidar.cpp` 中新增激光雷达数据处理独立模块，与相机处理解耦，便于独立调试
- 重构点云地图构建管线：支持增量式地图更新，每次扫描只更新变化的区域，建图耗时降低 80%

**效果**：定位精度从 20cm 提升至 5cm；点云配准达到实时（25fps）；地图支持在线增量更新。

### merge_ws39 — LiDAR 目标识别与上层决策

**背景**：之前激光雷达只用来定位和建图，没有做目标识别；视觉和激光的输出各管各的，没有统一决策层，冲突时无人仲裁。

**发现的问题**：
- 相机在光照变化（强光直射/阴影遮挡）下检测不稳定，但激光雷达不受光照影响——这种互补性没有被利用
- 视觉、激光、Z-buffer 三个感知通道各自输出检测结果，但缺乏仲裁机制，冲突时系统无法决策
- 激光雷达的点云数据只用来做配准，没有做目标级检测，浪费了激光的感知能力

**解决方案**：
- **新增 LiDAR 目标识别**（`lidar/lidar_recognition.h`, `lidar/lidar_getttf.h`）：对激光点云进行欧几里得聚类，提取每个聚类的尺寸、形状、反射强度等特征，识别出方块、机器人、障碍物等目标，与视觉检测做交叉验证
- **新增上层决策层**（`superstratum/super.h`, `super2.h`）：统一管理三个感知通道的输出，实现三级仲裁机制：
  1. 三通道一致 → 直接采纳，置信度设为 0.95
  2. 两通道一致 → 投票决定，置信度设为 0.85
  3. 全部冲突 → 取置信度最高的，置信度设为 0.70
- **新增数据滤波模块**（`filter.h`, `fliter.cpp`）：对来自不同传感器的数据进行时间戳同步（插值对齐）和空间滤波（剔除离群点），确保融合时数据在同一时刻、同一坐标系下
- 重构 `camera.cpp` 和 `lidar.cpp` 的接口，定义统一的 `detect()` 虚函数，新增传感器只需继承实现即可接入

**效果**：光照突变场景下检测召回率提升 15%；三通道互补使整体 F1 分数提升至 0.92。

### merge_ws39.5 — ORB 特征匹配与后处理增强

**背景**：视觉定位在纹理贫乏场景（纯色墙面、地毯）下容易丢失；决策融合后的输出存在异常跳动，无法直接用于控制。

**发现的问题**：
- 在 RoboMaster 赛场的白色围墙区域，视觉里程计因缺少特征点而失效，定位完全依赖轮式里程计
- 融合决策在目标切换时输出抖动明显（前后两帧的检测结果在空间上突变 10cm+）
- 遮挡处理在连续遮挡场景下不够稳定，频繁触发误遮挡

**解决方案**：
- **新增 ORB 特征模块**（`orb/orb_debug.h`, `orb_debug.cpp`）：在视觉定位中引入 ORB 特征提取与匹配。ORB 比 FAST+BRIEF 对光照变化更鲁棒，在纹理贫乏区域仍能通过角点检测维持定位
- **新增上层后处理层**（`superstratum/super_post.h`）：在决策融合后增加两级后处理：
  1. 卡尔曼平滑：对目标位置做时序平滑，消除单帧跳动
  2. 异常值过滤：若某帧的检测结果与预测值偏差 > 3σ，判定为异常并丢弃
  3. 输出限幅：防止单帧输出突变导致机械臂抖动
- 优化 `occlusion_handing`：引入连续帧的遮挡历史（滑动窗口 5 帧），减少单帧误判
- 优化 `vision.cpp`：整合 ORB 特征辅助定位，实现定位与检测双通道并行，即使检测通道故障也不影响定位

**效果**：纹理贫乏场景下视觉定位成功率提升 40%；融合输出抖动从 ±10cm 降至 ±2cm。

### merge_ws39.6 — 感知与控制管线全面优化

**背景**：系统在长时间运行（>30 分钟）后出现性能衰减：丢帧、控制抖动、坐标漂移，影响竞赛表现。

**发现的问题**：
- 相机的图像采集使用单缓冲，在高速运动时处理跟不上，丢帧率高达 15%
- 控制模块的 PID 参数在赛场的不同区域（平地/坡道/地毯）需要不同参数，但只有一个固定值
- 坐标变换在多帧累积后出现微小漂移，长时间后误差累计到不可忽略的程度
- 各模块的日志输出杂乱无章，难以定位问题

**解决方案**：
- **相机模块优化**（`camera.cpp/h`）：增加环形缓冲区缓存 30 帧，当处理延时高时仍能获取最近的帧；新增丢帧检测告警
- **控制模块优化**（`control.cpp`）：根据轮子打滑程度（通过里程计 vs 视觉里程计的差异判断）自适应切换三组 PID 参数（平地/坡道/地毯）
- **坐标变换重构**（`coordinate.h`）：增加异常坐标系检测机制——当变换矩阵的行列式偏离 1 超过阈值时，自动回退到上一帧的变换
- **主融合逻辑优化**（`merge.cpp`）：优化多传感器融合时序，使用时间戳对齐代替顺序处理，减少因数据不同步导致的融合误差

**效果**：30 分钟长时间运行测试中，丢帧率从 15% 降至 2%；控制抖动减少 60%。

### merge_ws39.7 — PnP 位姿估计与参数配置系统

**背景**：机械臂抓取方块时需要精确的 6D 位姿（X, Y, Z, Roll, Pitch, Yaw），但现有检测只能输出 2D 包围框；大量硬编码参数导致上车调试效率极低。

**发现的问题**：
- 之前的检测只输出 2D 坐标（像素 u, v），无法直接用于机械臂抓取，需要额外的 PnP 解算
- 每次参数调整（如 PnP 阈值、滤波参数、ROI 大小）都需要重新编译，调试一次耗时 5 分钟，严重拖慢进度
- Z-buffer 参数在真实环境中与仿真差异大，需要重新标定但缺乏工具

**解决方案**：
- **新增 PnP 位姿估计**（`PnP/pnp_main.h`, `pnp_func.h`, `pnp_param.h`, `pnp_debug.h`）：通过 2D 检测框角点与 3D 模型角点的对应关系，使用 PnP 算法解算目标 6D 位姿，直接输出用于机械臂抓取的 XYZ 坐标和朝向角
- **新增参数配置系统**（`parameter/config.yaml`, `parameter.cpp/h`）：将全部可调参数外部化到 YAML 文件，通过 ROS dynamic_reconfigure 支持运行时动态调整，无需重新编译
- **快速标定工具**（`calibration/fast_camera_calibration.h`）：简化外参标定流程，从原来需要 30 分钟的棋盘格标定缩短到 5 分钟，只需对着已知尺寸的方块拍摄几张照片即可
- **新增 R1/R2 控制模块**（`superstratum/controlR1.h`, `controlR2.h`）：为两种不同底盘车型提供差异化的运动控制策略（阿克曼 vs 全向轮）
- **新增 RC-EKF**（`rcekf/rcekf.h`, `predicted.h`）：鲁棒卡尔曼滤波器，在传感器异常（如相机掉线）时自动降级到仅用激光雷达定位

**效果**：目标定位从 2D (u,v) 升级到 6D (X,Y,Z,Roll,Pitch,Yaw)；参数调整无需重新编译，调试效率提升 10 倍。

### merge_ws39.75 — 感知管线细化与稳定化

**背景**：merge_ws39.7 虽然功能丰富了，但系统在长时间运行中的稳定性还需要加固。

**发现的问题**：
- 长时间运行后各模块出现参数累积漂移，检测精度逐渐下降
- 部分场景下坐标变换异常导致系统崩溃（segment fault）
- USB 相机在长时间运行后掉线，且不会自动重连
- 实际比赛场景中需要 USB 相机和 RealSense 相机同时工作

**解决方案**：
- 细化了 `camera/control/lidar` 各模块的运行时参数，定期自动重置累积状态
- 加固坐标变换异常检测与自动恢复机制
- 新增 USB 相机掉线自动重连逻辑，最多重试 5 次
- 新增 `usb_cam_multi.cpp` 支持双 USB 相机同时工作
- 新增 `apriltag/apriltag.cpp`, `apriltag_pose_module.h` 等 Apriltag 检测模块作为辅助定位，在视觉里程计失效时提供备份
- 新增 `mapping/mapping.cpp/h` 建图模块，支持在线保存和加载地图
- 新增 `debug/debug_vision.cpp`, `ros_debug.cpp/h` 统一调试输出格式

**效果**：系统 MTBF（平均无故障时间）从 30 分钟提升到 2 小时以上。

### merge_ws39.8 — 上位机集成与最终优化

**背景**：竞赛需要上位机（场外电脑）远程监控与控制，全链路需要在真实比赛场景中做最终验证。

**发现的问题**：
- 工控机在机器人上，调试人员在场外无法实时查看机器人状态

- KFS 方块检测器需要集成到融合管线中，但尚未完成

**解决方案**：
- **新增上位机交互**（`superstratum/super_post.h`）：实现工控机与上位机的双向数据通信，通过 ROS Topic 传输状态信息，上位机可远程启停任务
- **新增 KFS 检测器**（`kfs_detector/kfs_detector.cpp`, `kfs_detector.h`）：集成 KFS 方块检测算法，统一管理所有目标的检测、跟踪与输出
- 模型文件管理：新增 `model/han/` 目录，规范 pt/onnx/xml/yaml 文件的存储结构
- 标定工具升级：支持在线标定，无需停止系统即可微调外参
- 确定多传感器融合最终权重与决策逻辑，通过 3 次完整模拟比赛验证

**效果**：上位机可实时查看检测状态和控制机器人；全链路通过竞赛场景验收。

### rc_ws1 — 竞赛候选版

**背景**：系统需要通过竞赛级别的可靠性验证，修复集成测试中发现的边界条件。

**发现的问题**：
- 极端光照条件下（太阳直射镜头）检测退化
- 多机器人协同时的通信冲突
- 长时间运行后的内存泄漏（约 2.5 小时后发生）

**解决方案**：
- 全链路系统集成测试，修复了 20+ 边缘 Case

### rc_ws2 — 竞赛最终版

**背景**：竞赛需要设备自检、双雷达热备、挑战赛路径管理等运行期保障功能，确保系统在比赛当天不翻车。

**发现的问题**：
- 没有开机自检机制，竞赛开始时才发现某个传感器坏了
- 单激光雷达故障时缺乏备份方案，会导致定位完全丢失
- 挑战赛需要预设路径自动导航，但之前都是手动操控
- 队友上手困难，缺乏系统性文档

**解决方案**：
- **新增设备自检系统**（`selfinspection/inspection.h`）：启动时自动检测相机（是否能打开、是否有图像）、激光雷达（是否有点云）、IMU（是否有数据），结果通过上位机显示，异常时声光告警
- **双雷达支持**：新增主/备雷达切换逻辑，主雷达故障时 500ms 内自动切换到备用雷达
- **挑战赛路径管理**（`shell_monitor2/challenge_path/`）：预设 20+ 条竞赛路径，覆盖红蓝半场、不同赛题配置，支持一键切换
- **竞赛文档体系**（`doc/`）：包含 9 篇系统架构和模块使用文档，覆盖串口通信、单例模式、Call_once、智能指针等核心知识点，方便队友快速上手
- **控制中心升级**（`control_center.py`）：集成设备自检、参数配置、任务队列、红蓝半场切换、挑战赛模式选择等功能，全部通过图形界面操作
- **地图持久化**（`map/last`）：支持断电后快速恢复最近使用的地图，无需重新建图
- **模型文件管理**：双色模型（blue/red）分类存储，比赛时根据阵营自动选择对应模型

**效果**：系统通过最终竞赛验证，成功完成挑战赛全部任务，运行期间无故障。
